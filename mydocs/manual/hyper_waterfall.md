# Hyper-Waterfall: AI 시대의 소프트웨어 개발 방법론

> AI가 없던 시대에는 불가능했던 방법론.
> 워터폴의 규율과 애자일의 속도를 동시에.

## 1. 왜 새로운 방법론이 필요한가

### 워터폴의 실패

워터폴은 올바른 철학을 가지고 있었다 — 계획하고, 문서화하고, 검증하라.
하지만 **속도가 너무 느렸다.** 시장은 기다려주지 않았다.

### 애자일의 타협

속도를 얻었지만 대가가 있었다:
- **문서가 사라졌다** — "코드가 곧 문서"라는 변명
- **방향이 흔들렸다** — 스프린트마다 우선순위가 바뀜
- **지식이 사람 머리에 갇혔다** — 핵심 개발자가 떠나면 프로젝트가 멈춤

### 바이브 코딩의 함정

AI 출력을 읽지도 않고 수락한다. 겉보기에는 빠르지만,
**이해하지 못한 코드는 고칠 수도 없다.**

## 2. Hyper-Waterfall 정의

**거시적으로는 워터폴, 미시적으로는 애자일 — AI가 이 둘을 동시에 가능하게 한다.**

```
거시 (프로젝트 수준) — 워터폴의 규율:
  계획 ──→ 설계 ──→ 구현 ──→ 검증 ──→ 배포
  문서화   문서화   문서화   문서화   문서화

미시 (타스크 수준, 수 시간) — 애자일의 속도:
  구현 → 테스트 → 피드백 → 수정 → 테스트 → ...
   AI     자동화   사람 판단   AI    자동화
```

| | 전통 워터폴 | 애자일 | 바이브 코딩 | Hyper-Waterfall |
|--|-----------|--------|-----------|-----------------| 
| 규율 | ●●●●● | ●●○○○ | ○○○○○ | ●●●●● |
| 속도 | ●○○○○ | ●●●○○ | ●●●●● | ●●●●● |
| 문서 | ●●●●● | ●○○○○ | ○○○○○ | ●●●●● |
| 전제조건 | 대규모 팀 | 소규모 팀 | AI 도구 | **AI + 숙련된 사람** |

## 3. 핵심 원칙

### 제 1원칙: 구현 가치(목표)가 AI 컨텍스트에 유지되도록 한다

- **CLAUDE.md**에 프로젝트 규칙, 빌드 방법, 워크플로우를 명시
- **타스크 등록 시** 목표와 범위를 명확히 기술
- **컨텍스트가 길어지면** 핵심 맥락을 요약하여 재주입

### 제 2원칙: 작업 통제권을 작업지시자가 유지해야만 한다

- **계획은 AI가 제안하되, 승인은 사람이 한다**
- **AI가 임의로 작업 종료를 제안하지 않는다** — 작업지시자가 결정
- **아키텍처 변경은 반드시 사람의 판단을 거친다**

### 제 3원칙: 주기적으로 AI 컨텍스트 유지를 확인해야 한다

- **핵심 결정사항을 문서로 외부화** — AI 메모리가 아닌 파일에 기록
- **새 세션 시작 시** 이전 맥락을 요약하여 전달
- 컨텍스트 압축 시점을 인지 — AI가 이전 결정을 잊기 시작하면 경고 신호

> 이 세 원칙은 "사람은 절대 생각을 멈추지 않는다"의 구체적 실천이다.

## 4. 역할 분담

### 작업지시자 (사람)

사람은 **생각하는 역할**에 집중한다:
- 방향 설정: "다음에 뭘 해야 하는가?"
- 우선순위: "무엇이 더 중요한가?"
- 품질 판단: "이것이 충분히 좋은가?"
- 아키텍처 결정: "이 AWS 구조가 올바른가?"
- 도메인 지식: "이 크롤링 소스는 어떻게 동작하는가?"

### AI (Claude Code)

AI는 **실행하는 역할**에 집중한다:
- 분석: 코드베이스 탐색, 원인 추적
- 계획: 수행·구현 계획서 작성
- 구현: Lambda 함수, CDK 스택, React 컴포넌트 작성
- 문서: 단계별 보고서, 기술 문서, 커밋 메시지 작성

> **사람은 절대 생각을 멈추지 않는다.**

## 5. 타스크 사이클

```
1. 타스크 등록
   └─ 작업지시자: GitHub Issue 생성, 범위 정의

2. 수행 계획서
   └─ AI: plans/task_{m}_{N}.md 작성
   └─ 작업지시자: 검토 → 승인 또는 수정 요청

3. 구현 계획서
   └─ AI: plans/task_{m}_{N}_impl.md 작성 (최소 3단계, 최대 6단계)
   └─ 작업지시자: 승인

4. 단계별 구현
   └─ AI: 코드 작성 + 테스트 + working/task_{m}_{N}_stage{N}.md 작성
   └─ 작업지시자: 검증 → 승인 또는 피드백

5. 최종 보고
   └─ AI: report/task_{m}_{N}_report.md 작성
   └─ 작업지시자: 승인 → 이슈 클로즈 → merge
```

## 6. 품질 관문

모든 타스크는 다음을 통과해야 한다:

1. **계획 승인** — 구현 전에 방향이 맞는지 확인
2. **테스트 통과** — `npm run test -ws`, ESLint, TypeScript 타입 검사
3. **인프라 검증** — `cdk diff`로 의도치 않은 리소스 변경 없음 확인
4. **기능 검증** — SAM Local 또는 dev 스택에서 직접 동작 확인
5. **회귀 방지** — 기존 API 엔드포인트 응답 스펙 변경 없음

## 7. 이 방법론이 서브컬쳐 트래커 프로젝트에 적합한 이유

- **복잡한 도메인**: 크롤링 소스별 HTML 구조, AWS 서비스 연동, Bedrock 프롬프트 설계
- **높은 품질 기준**: 실시간 알림, 데이터 중복 방지, 크롤링 안정성
- **문서화 필요**: 크롤러 소스 추가 절차, API 스펙, DB 스키마가 명확히 문서화되어야 함
- **1인 또는 소규모 팀**: 모든 결정과 디버깅 과정이 문서에 남아야 나중에 참조 가능
